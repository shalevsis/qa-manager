# Internationalization (i18n) Test Suite

## When to apply
Use when a feature renders user-visible text, dates, numbers, or currency, or when locale switching is supported. Run for every new string added to the UI.

## What to test
- Every user-visible string is sourced from a translation key (no hardcoded English literals)
- A missing translation key renders a fallback (key name, default string, or empty) — not a crash
- RTL locales (Arabic, Hebrew) mirror layout: padding/margin sides, text-align, icon positions
- Dates are formatted using the active locale (e.g. MM/DD vs DD/MM vs locale-specific calendar)
- Numbers and currency respect locale separators and symbol position
- Switching locale at runtime updates all displayed strings without requiring a page reload
- Pluralization rules are applied correctly (1 item vs 2 items vs 0 items)
- Long translations (German, Finnish) do not overflow containers or truncate incorrectly
- Interpolated values (user name, counts) are inserted in the correct position per locale

## Key patterns

**No hardcoded strings**
```
// Fail: text directly in markup
render(<p>Submit</p>)

// Pass: sourced from translation
render(<p>{t('actions.submit')}</p>)
// Assert: rendered text matches translation fixture, not a literal
```

**Missing key fallback**
```
// Remove key from translation map
delete translations['en']['actions.submit']
render(<Button />)
// Assert: no throw, renders key name or default, not blank screen
```

**RTL layout**
```
setLocale('ar')
const el = screen.getByTestId('card')
const styles = getComputedStyle(el)
assert styles.paddingInlineStart === styles.paddingRight  // logical vs physical
assert styles.direction === 'rtl'
```

**Pluralization**
```
for (const [count, expected] of [[0,'items'],[1,'item'],[2,'items']]) {
  assert t('cart.items', { count }) === expected
}
```

**Date/number locale**
```
setLocale('de-DE')
assert formatNumber(1234567.89) === '1.234.567,89'
assert formatDate(new Date('2025-01-15')) === '15.01.2025'
```

## Common gaps
- Tests only run in the default locale — RTL and non-Latin locales never exercised
- Missing key silently returns `undefined`, which React renders as nothing (visually broken but no error thrown)
- Pluralization tested only for count=1 and count=2, missing count=0 and language-specific rules (e.g. Slavic languages have 3+ plural forms)
- Date formatting tested with hardcoded expected string instead of `Intl.DateTimeFormat` — breaks when locale library is updated
- Locale switch test re-renders but asserts on stale DOM snapshot before update flushes
- Long-string overflow only checked visually in screenshots, not asserted programmatically (e.g. `scrollWidth > offsetWidth`)
